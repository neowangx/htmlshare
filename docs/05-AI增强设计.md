# 05 AI 增强设计

> **⚠️ 已被 D18 取代（2026-07-11）**：增强层已从 v1（template/style/slots）整体切换为 **A2UI 静态子集**。当前权威来源是 **docs/11（A2UI 静态渲染重构）+ docs/04 §4/§8 + SKILL.md「Enhancement Rules」**。本文档以下的模板/槽位/rubric 规则仅作历史参考，不再执行。红线（不改事实、不丢信息、白名单净化）仍然有效。
> 必读前置：docs/00（D5/D6/**D18**）、docs/04（§4 增强片段契约、§8 枚举）、docs/11。
> 本文原为 SKILL.md 增强部分的内容母版。增强由宿主 agent 执行，无独立 LLM API 调用（D6）。

## 1. 增强的目标与红线

**目标**：让读者更快抓住重点、更好理解。允许重写措辞、精简冗余、重组顺序。
**红线**（违反任一条即为不合格产出）：
1. 不得改变事实：数字、日期、金额、人名、结论、承诺一字不错。
2. 不得丢信息：原文每个信息点必须在增强版可找到（可在折叠区）。
3. 不得添加原文没有的观点或推断（TL;DR 只能提炼，不能引申）。
4. 忠实版永远同时存在，一键可切（组装由 CLI 保证）。

## 2. 内容类型分类 rubric（选模板用）

按信号从上到下匹配，**首个命中即选定**；都不中 → `generic`：

| 模板 | 判定信号（任意 2 条命中即成立） |
|------|------------------------------|
| meeting | 有与会人/时间地点；有决议或行动项句式（"决定/负责/截止"）；按发言或议题推进；标题含"会议/纪要/评审/同步" |
| proposal | 提出问题并给方案；有"目标/方案/计划/预算/风险"类小节；面向说服决策者 |
| tutorial | 按步骤编号推进；有前置条件/环境要求；祈使句为主（"运行/点击/安装"） |
| release | 版本号/日期开头；变更列表（新增/修复/破坏性变更）；升级说明 |
| generic | 以上皆非（散文、分析、笔记、混合体） |

**置信度低时（信号 <2 条）必须选 generic**——错配模板比不用模板伤害更大。

## 3. 风格选择规则

| 选择 | 适用信号 |
|------|---------|
| clinical（临床卡片，默认） | 商务/客户场合、纪要、方案——克制专业，冷调卡片 |
| minimal（极简黑白） | 纯文字长文、分析、随笔——排版即全部 |
| editorial（杂志编辑） | 面向公开阅读的内容、教程、发布公告——有标题感与节奏 |
| darktech（暗色科技） | 代码密集、开发者受众、原型/demo 配套说明 |

用户在 config/`--style` 指定则**无条件服从**；未指定时按上表选，拿不准选 clinical。

## 4. 增强规则（写进 SKILL.md 的操作指令）

1. **TL;DR（1~5 条）**：只提炼「读者最需要带走的」——结论 > 待办 > 关键数据。每条 ≤ 40 字，可读性优先于完整性（完整性由正文负责）。
2. **填槽**：按所选模板的槽位集（docs/04 §8）重组内容；无对应内容的槽位**省略**（不得编造凑数）。
3. **交互组件使用条件**（不满足条件就不用——克制是准则，docs/02 §1）：
   - `<details>` 折叠：仅用于「过程性/背景性内容」（讨论过程、原始记录、附录），一级信息永不折叠。
   - 卡片分组：仅当条目 ≥3 且同构（如行动项列表）。
   - 表格：仅当确有二维数据；禁止把段落硬转表格。
4. **行动项特殊处理**（meeting/proposal）：提取为「负责人 + 事项 + 期限」结构，缺项标"未指定"，不得脑补。
5. **代码块、引用、链接原样保留**，不改写代码。
6. 产出写入 `enhanced.json`（docs/04 §4），交给 CLI：`htmlshare publish <file> --enhanced enhanced.json`。

## 5. 输出校验（CLI 端 compose.js 执行，确定性）

| # | 校验 | 失败处置 |
|---|------|---------|
| V1 | JSON 可解析、version=1、必填字段齐 | 重试 1 次（把校验错误反馈给 agent）→ 仍败走忠实版 |
| V2 | template/style ∈ docs/04 §8 枚举 | 同上 |
| V3 | slot ∈ 所选模板槽位集 | 剔除非法 section，其余继续 |
| V4 | sections[].html 过 sanitize 白名单（+details/summary），无 script/事件属性/外链资源 | 净化后继续；净化后为空则视为 V1 失败 |
| V5 | tldr 1~5 条、每条 ≤80 字符 | 截断/裁剪后继续 |
| V6 | 增强版总字符数 ≥ 原文的 30%（防增强版丢信息式偷懒） | 警告 + 走忠实版 |

## 6. 失败兜底表（与 docs/02 E1/E8 对应）

| 情形 | 兜底 |
|------|------|
| agent 未提供 enhanced.json（headless/弱模型/用户跳过） | 忠实版发布，输出注明「本次为原文版」 |
| V1/V2 校验失败且重试仍败 | 忠实版发布 + 一行原因 |
| agent 生成超时（用户等不及） | SKILL.md 指示：>60s 未产出即放弃增强直接发忠实版 |
| 用户 `--template/--style` 与内容明显错配 | 服从用户，不做二次判断 |

## 7. 回归测试样例（3 例，进 test/ 作为固定资产）

> 每例 = 输入 md + 「期望特征断言」清单。断言针对 CLI 确定性部分与 rubric 规则，可自动执行；agent 产出质量断言标注「需人工」。

**样例 A：会议纪要**（输入：含 3 个决议、4 个带负责人的行动项、两页讨论记录的 md）
- 断言：rubric 应判 meeting（信号：决议句式+议题推进）；enhanced.json 用 meeting 槽位；actions 含 4 项且负责人与原文一致（可 grep 断言）；讨论记录出现在 discussion 槽且被 `<details>` 包裹；V1~V6 全过；toggle 后忠实版与 convert.js 输出逐字节一致。

**样例 B：技术教程**（输入：带编号步骤、代码块、前置条件的 md）
- 断言：判 tutorial；代码块内容与原文逐字节一致（红线 5）；steps 槽位保持步骤顺序；无表格（原文无二维数据，克制规则）。

**样例 C：无结构随笔**（输入：2000 字散文，无小节）
- 断言：判 generic（置信度规则）；tldr ≤5 条；无折叠（无过程性内容）；V6 通过（不丢信息）。

**对抗样例 D（校验器测试，不走 agent）**：手工构造含 `<script>`、非法 slot、8 条 tldr、version=2 的 enhanced.json → 断言 V1~V5 各自的处置行为逐条触发。
